# Технічне завдання (ТЗ)

## 1. Загальна інформація

- **Назва проєкту:** DigitCA
- **Призначення:** надання сервісу Certificate Authority (CA) для ініціалізації PKI, випуску, перевірки, відкликання та публікації X.509 сертифікатів.
- **Формати роботи:**
  - CLI-команди
  - REST API (`/api/v1/...`)

## 2. Цілі системи

1. Забезпечити централізоване керування сертифікатами (Root/Intermediate/Leaf).
2. Забезпечити автентифікацію та авторизацію операторів через LDAP.
3. Забезпечити аудит дій операторів.
4. Забезпечити інтеграційний REST-контракт з OpenAPI/Swagger.
5. Забезпечити механізм ревокації через CRL.

## 3. Межі проєкту

### 3.1 У межах поточної версії

- Ініціалізація Root CA.
- Ініціалізація Intermediate CA.
- Випуск сертифікатів профілів `server-tls` та `client-auth`.
- Підтримка SAN: DNS і IP.
- Відкликання сертифікатів з типізованою причиною (`RevocationReason`).
- Генерація CRL:
  - `GET /crl/root.crl`
  - `GET /crl/intermediate.crl`
- Перевірка сертифікатів (`verify`).
- Аудит-лог у MongoDB.
- Публікація записів сертифікатів у LDAP (опційно).
- OpenAPI JSON + Swagger UI.

### 3.2 Поза межами поточної версії

- OCSP responder.
- Автоматична ротація CA ключів.
- Вбудований UI для адміністрування.
- Інтеграція з HSM/KMS на рівні драйверів.

## 4. Архітектура та компоненти

- **API-шар (Axum):** маршрути, DTO, auth extractor, middleware.
- **Service-шар:** бізнес-логіка CA операцій, аудит, валідація.
- **CA-шар (OpenSSL):** генерація ключів/сертифікатів/CRL, підпис.
- **Storage-шар:** MongoDB репозиторій + InMemory реалізація для тестів.
- **LDAP-шар:** authN/authZ та LDAP публікація.
- **Конфігурація:** `.env` + `AppConfig`.

## 5. Функціональні вимоги

### 5.1 Root та Intermediate

- Система має ініціалізувати Root CA.
- Система має ініціалізувати Intermediate CA на базі Root CA.
- Система має дозволяти експорт Root/Intermediate PEM.

### 5.2 Випуск сертифікатів

- Система має випускати сертифікати за параметрами:
  - `common_name`
  - `profile`
  - `issuer` (`auto`, `root`, `intermediate`)
  - `dns_names`
  - `ip_sans`
  - `validity_days`
- Для `server-tls` система має вимагати мінімум один SAN (`dns` або `ip`).
- Приватний ключ leaf-сертифіката має повертатися тільки в момент `issue`.

### 5.3 Ревокація та CRL

- Система має відкликати сертифікат за `serial`.
- Система має валідовувати причину ревокації.
- Система має формувати CRL з `reasonCode` для revoked entries.
- Система має розділяти CRL за issuer (root / intermediate).

### 5.4 Верифікація

- Система має перевіряти:
  - валідність підпису,
  - ревокацію,
  - часову валідність.

### 5.5 Аудит

- Система має писати аудит подій (actor, type, target, time, details).
- Система має повертати аудит-лог з обмеженням за `limit`.

### 5.6 LDAP

- Система має виконувати Basic auth через LDAP.
- Система має підтримувати LDAP-пошук сертифікатів за CN.
- Система має опційно публікувати/депублікувати записи сертифікатів у LDAP.

## 6. Нефункціональні вимоги

### 6.1 Безпека

- Basic auth має бути дозволений лише через HTTPS у production.
- CORS має бути обмежений whitelist-ом.
- Passphrase CA ключів має бути задана у production.
- Root/Intermediate приватні ключі мають зберігатися зашифрованими за наявності passphrase.

### 6.2 Надійність

- Сервіс має коректно обробляти помилки MongoDB/LDAP/crypto.
- Логіка має покриватися unit + integration тестами.

### 6.3 Спостережуваність

- Логи мають бути доступні через стандартний stdout/stderr потік.
- Має бути доступний health endpoint (`/health`).

## 7. REST API (мінімальний перелік)

- `GET /health` — перевірка доступності та базового стану сервісу.
- `POST /api/v1/ca/root` — ініціалізація Root CA (генерація ключа і самопідписаного сертифіката).
- `GET /api/v1/ca/root` — експорт Root CA сертифіката у PEM.
- `POST /api/v1/ca/intermediate` — ініціалізація Intermediate CA, підписаного Root CA.
- `GET /api/v1/ca/intermediate` — експорт Intermediate CA сертифіката у PEM.
- `POST /api/v1/certificates` — випуск leaf-сертифіката за вхідними параметрами профілю/SAN/issuer.
- `GET /api/v1/certificates` — отримання списку сертифікатів з фільтрацією та пагінацією.
- `GET /api/v1/certificates/{serial}` — отримання детальної інформації про сертифікат за serial.
- `POST /api/v1/certificates/{serial}/revoke` — відкликання сертифіката із вказанням причини ревокації.
- `GET /api/v1/certificates/{serial}/verify` — перевірка підпису, статусу ревокації та валідності по часу.
- `GET /api/v1/certificates/{serial}/chain` — експорт повного PEM-ланцюжка довіри для сертифіката.
- `GET /crl/root.crl` — отримання CRL для сертифікатів, виданих Root CA.
- `GET /crl/intermediate.crl` — отримання CRL для сертифікатів, виданих Intermediate CA.
- `GET /api/v1/audit` — отримання журналу аудиту операцій CA.
- `GET /api/v1/ldap/certificates?cn=...` — пошук сертифікатів у LDAP-каталозі за Common Name.

## 8. Критерії приймання

1. `cargo test` проходить без падінь.
2. Swagger UI доступний за `/docs`, OpenAPI JSON за `/api-doc/openapi.json`.
3. Випуск leaf-сертифіката працює, `key_pem` повертається лише при issue.
4. Ревокація з валідною reason працює; з невалідною повертає 422.
5. Root/Intermediate CRL формуються і містять revoked entries з reasonCode.
6. При включеному HTTPS enforcement запити Basic auth без HTTPS-ознаки відхиляються.

## 9. Ризики та припущення

- Для production потрібні зовнішні MongoDB та LDAP з резервуванням.
- Для production потрібен reverse proxy (TLS termination).
- Без HSM/KMS зберігання ключів покладається на безпеку ОС та секретів.
